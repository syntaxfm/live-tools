### Jazz Data Model

- Jazz is the application database and sync layer. All app data the client reads or mutates goes through Jazz.
- Schema lives in `src/lib/schema.ts`. The Better Auth tables are merged in via `...betterAuthSchema` from `src/lib/schema-better-auth/schema.ts` (generated by `pnpm auth:schema`).
- App tables today: `shows`, `showHosts`, `hostLinks`, `tickerMessages`, `audienceSubmissions`, `submissionVotes`. Better Auth tables: `better_auth_user`, `better_auth_session`, `better_auth_account`, `better_auth_verification`, `better_auth_jwks`.
- The app user is `better_auth_user` directly. There is no separate `appUsers` table. References to a user use `s.ref('better_auth_user')`.
- Permission rules live in `src/lib/permissions.ts`.
- Queries and mutations belong **inline in the component** that owns the view (see Querying / Mutating sections below). Feature folders under `src/lib/components/<feature>/` exist for components, types, and the rare multi-step orchestration helper — not for one-line `db.*` wrappers.
- After schema or permission changes run `pnpm auth:validate`. After any TS/Svelte change run `pnpm check`.
- Typed row/insert helpers are exported from `src/lib/schema.ts` (`Show`, `ShowInsert`, `AudienceSubmission`, etc.). Import these instead of redefining shapes.

### Querying Data

- Construct `QuerySubscription` from `jazz-tools/svelte` **inline in the component** that owns the view. Do not relocate the query into a separate helper just to keep the component short.
- Query through the typed app: `app.audienceSubmissions.where({ showId: show.id })`.
- Return `undefined` from the subscription factory to skip the query until required ids exist:

  ```ts
  const submissions = new QuerySubscription(() =>
  	show.id ? app.audienceSubmissions.where({ showId: show.id }) : undefined
  );
  ```

- Do not show loading states — just check whether the data exists. Use `subscription.current ?? []` for list UI.
- Surface `subscription.error` when present (see `AdminSubmissionQueue.svelte`).
- Get the current user with `getSession()` and read claims directly in the component:

  ```ts
  const session = getSession();
  const is_admin = $derived(session?.claims.is_admin === true);
  ```

- Do not extract `*.svelte.ts` query helpers as a default. Lift a query factory only when **multiple components** need the exact same query construction — and even then, prefer the simpler option of repeating the one-liner.
- Do not read client app state from server `load` data when that state belongs in Jazz. Server `load` is for static route setup or non-Jazz boundaries.

### Mutating Data

- Write mutations **inline in the component event handler**. Grab `const db = getDb();` once at the top of the component and call `db.insert` / `db.update` / `db.delete` directly. Do not extract a helper function just to pass `db` through — that's hot-potatoing state across module boundaries for no benefit.

  ```svelte
  <script lang="ts">
  	import { getDb } from 'jazz-tools/svelte';
  	import { app, type AudienceSubmission } from '$lib/schema';

  	let { submission }: { submission: AudienceSubmission } = $props();
  	const db = getDb();

  	function approve() {
  		db.update(app.audienceSubmissions, submission.id, { status: 'approved' });
  	}

  	function remove() {
  		db.delete(app.audienceSubmissions, submission.id);
  	}
  </script>

  <button onclick={approve}>Approve</button>
  <button onclick={remove}>Delete</button>
  ```

- **Authorization is enforced by Jazz permissions** in `src/lib/permissions.ts`. The permission layer rejects writes the session isn't allowed to make. Do not wrap mutations in JS `assertAdmin` guards — that's redundant whether the mutation is inline or inside a `*-actions.ts` helper. UI-side `is_admin` derived values exist to show/hide affordances, not to authorize the write. JS-side invariant checks that the permission layer does _not_ express (e.g. "submission must be `approved` before featuring") are still legitimate.
- Treat Jazz writes as local-first. Use `.wait({ tier: 'global' })` only when the next step or UI actually needs confirmed global sync (e.g. you're about to navigate, or chain another write against the inserted id). Fire-and-forget UI writes don't need it.
- Only extract a helper into a `*-actions.ts` file when the operation is **non-trivial multi-step orchestration** — for example the cascading delete in `show-actions.ts` `deleteShow`, which reads from five tables, deletes children in batched `Promise.all`s, then deletes the parent. A single `db.update` is never that.
- For cascading deletes, batch reads with `db.all(..., { tier: 'global' })`, then run each table's deletes in `Promise.all` with `.wait({ tier: 'global' })` per row. See `deleteShow` for the canonical shape.
- Never add REST endpoints or SvelteKit form actions for ordinary app data mutations when Jazz can perform the write directly.
- API routes are for true server-only boundaries: auth/JWT bridge, secrets, external APIs, server-owned integrations (today only `src/routes/api/page-title/+server.ts`).

### Auth and Roles

- App state and identity are Jazz-first. Components read the current app user from the Jazz session, not Better Auth session data.
- Better Auth is only the auth bridge: GitHub sign-in/sign-out, issuing the JWT, and exposing JWKS for Jazz JWT verification. Its tables live in the same Jazz DB via `jazzAdapter` (`src/lib/server/auth.ts`).
- The Jazz client is created in `src/routes/+layout.svelte` with the Better Auth JWT — there is no separate `JazzAuthProvider` component:

  ```svelte
  const jwt = await authClient.token();
  client = createJazzClient({
      env: dev ? 'dev' : 'prod',
      appId: env.PUBLIC_JAZZ_APP_ID,
      jwtToken: jwt?.data?.token ?? undefined,
      serverUrl: env.PUBLIC_JAZZ_SERVER_URL || 'http://localhost:7012/'
  });
  ```

- Client-side session data comes from `getSession()` (or `getJazzContext().session`).
- Profile fields (`name`, `email`, `image`, `githubUsername`) live on `better_auth_user`. Read them through the Jazz session claims for the signed-in user, or query `app.better_auth_user` for other users.
- Admin status is **not** stored as a mutable DB field — do not gate admin UI or Jazz permissions off a role column on a user row that a user could edit.
- Admin source of truth: the `ADMIN_GITHUB_USERNAMES` env var (comma/space-separated). On sign-in, `mapProfileToUser` in `src/lib/server/auth.ts` calls `isAdminGithubUser` (`src/lib/server/github-auth.ts`) against the GitHub login and writes `roles: ['viewer', 'admin', 'host']` for admins, otherwise `['viewer']`.
- The Better Auth JWT `definePayload` derives `claims.is_admin` from `user.roles.includes('admin')`. The full claims set is `{ id, name, email, image, githubUsername, is_admin }`.
- Jazz permissions enforce admin access with the signed claim: `session.where({ 'claims.is_admin': true })` (`is_admin` constant in `src/lib/permissions.ts`).
- UI labels may display admin/viewer derived from `session?.claims.is_admin`, but the authoritative check is always the signed Jazz session claim.
- Better Auth's internal tables (`better_auth_session`, `better_auth_account`, `better_auth_verification`, `better_auth_jwks`) are locked down with `allowRead/Insert/Update/Delete.never()` in permissions. Do not loosen these.
- When debugging a user showing as non-admin, check (1) the JWT claims on the Jazz session, (2) whether the GitHub login matches `ADMIN_GITHUB_USERNAMES`, (3) `user.roles` on the `better_auth_user` row.

### SvelteKit Patterns

- Routes are organized into groups: `(app)/(protected)` for authenticated app UI, `(auth)` for login, `(overlay)` for transparent OBS browser-source overlays, `(public)` for unauthenticated pages.
- Use nested SvelteKit layouts for shared route UI instead of thin wrapper components.
- Components own their own data. Read `getDb()`, `getSession()`, and `QuerySubscription` directly in the component that owns that view. Do not prop-drill route params or Jazz state through pass-through components.
- Props are for real generic boundaries (e.g. `let { show }: { show: Show } = $props();`). Manager imports are for feature/state boundaries.
- Use Svelte 5 runes in component state. `$state` for local reactive state, `$derived` for computed values. Do not use `$derived`/`$derived.by` for rename-only aliases of existing values.
- Avoid `$effect` unless you absolutely have to — none of the existing components use it.
- Event handlers: `onclick={handler}` or `onclick={() => handler(id)}` for capturing an argument. Do not write `onclick={() => { void handler(); }}`.
- Form/input handlers (`onsubmit`, `onchange`, `oninput`) extract values via the typed `event.currentTarget` (with `instanceof` narrowing where the value comes from the DOM). See `ShowStatePanel.svelte` for the pattern.
- Type-safe overlay routing: extend `OverlayFeature` in `src/lib/utils/overlays.ts` and add to `OVERLAY_FEATURES`. The `OverlayFeaturePath` template-literal type keeps paths in sync.
- The root `+layout.svelte` awaits the Better Auth token at the top level before creating the Jazz client. Do not reintroduce SSR — `+layout.ts` exports `ssr = false` for a reason.

### Styling

- The visual system is custom CSS in `src/app.css` with CSS custom properties (`--color-accent`, `--color-danger`, `--color-success`, `--radius-md`, `--shadow-raised-md`, `--transition-fast`, etc.) and Space Grotesk / Space Mono fonts.
- Use **data attributes** for variants, not class proliferation: `data-depth="medium"`, `data-state="warning"`, `data-variant="danger"`, `data-active="true"`. Selectors in component scoped styles target these (`section[data-depth='medium']`, `li[data-active='true']`).
- Component-scoped `<style>` blocks own their layout. Keep structural CSS (grid, flex, sizing, spacing) in the component; design tokens in `app.css`.
- Overlays render at 1920×1080 with transparent background via `html:has(.overlay-root)` in `app.css`. Overlay components live in `src/lib/components/shows/*Overlay.svelte` and routes under `(overlay)/overlay/<feature>/`.
- Do not add a `button` class to a `<button>` element. Style buttons via `data-variant` or scoped selectors.

### Utilities and Feature Colocation

- All shared/pure utilities live in `src/lib/utils/`. One pool — do not create new utility folders.
- Current utils: `github-usernames.ts`, `lower-thirds.ts`, `overlays.ts`, `page-title.ts`, `shows.ts`, `submissions.ts`, `ticker-messages.ts`, `urls.ts`. Check here first and extend an existing file before adding a new one.
- Server-only helpers go in `src/lib/server/` (`auth.ts`, `auth-jazz-context.ts`, `github-auth.ts`, `page-title.ts`).
- Feature-specific stores, action files, types, components, and (eventually) tests stay together under `src/lib/components/<feature>/`. Do not place feature-specific files in global buckets.
- Never write a utility function inline in the file where it is used if it is generic/pure. Lift it to `src/lib/utils/`.

### TypeScript and Code Style

- `tsconfig.json` has `strict: true`, `allowJs: true`, `checkJs: true`. Preserve strict typing — no `any`, prefer `unknown` at trust boundaries, narrow before access.
- Use `import type` for type-only imports. Group imports as: node built-ins, external packages, internal modules (`$lib/...`, `$env/...`, `$app/...`).
- Never re-export code or types; always import from the source module.
- Naming: `PascalCase` components/types, `camelCase` variables/functions, `SCREAMING_SNAKE_CASE` top-level constants, `is*`/`has*`/`should*` for booleans (the codebase uses `is_admin` with underscore to mirror the Jazz claim name).
- Add explicit return types on exported functions (`Promise<Show>`, `Promise<void>`, etc.).
- Error handling: log with context (`console.error('Unable to ...', error)`) and surface a user-facing message via local `$state`. Throw early on bad input with a clear message.
- Action files use option-object parameters (`interface CreateShowOptions { ... }`) rather than positional arguments.

### Testing

- There are no tests yet in this repo. When adding tests, colocate them with the feature (`*.test.ts` / `*.spec.ts` in `src/lib/components/<feature>/` or next to the util). Use `@testing-library/svelte` for component tests. Prefer role/text-based assertions. Keep tests deterministic and independent.

### Operational Rules

- **ALWAYS ASK BEFORE UPDATING the schema.** `src/lib/schema.ts` and `src/lib/permissions.ts` changes require confirmation.
- **Do not start additional Jazz dev processes** — use the already running one. The Vite/Jazz dev server runs on port 7012; the app on 7011.
- Do not run `pnpm db:reset` without asking — it wipes the local Jazz dev cache.
- After auth changes, run `pnpm auth:schema` to regenerate `src/lib/schema-better-auth/schema.ts`. After schema or permission edits run `pnpm auth:validate`. After TS/Svelte edits run `pnpm check`.
